# Introduction to PIX Instant Payments

Summarize

This functionality describes handling of PIX Instant Payments in Transact that covers outgoing (same-day and future-dated), incoming, and intrabank PIX transactions. It outlines the API-based initiation, validation checks, fund handling, interaction with external payment systems, and the resulting transaction statuses and accounting entries.

## PIX Outgoing Payments

This processing covers the following scenarios:

- Same-Day PIX Outgoing (PIXOUT)

  Banco Central do Brasil introduced PIX as Brazil’s Instant Payments System (SPI), enabling real-time fund transfers 24×7×365. The flow of this scenario is given below:

  1. Payment Order processes same-day PIX Outgoing payments through the Generic API (v7.4.0) using PAYMENT.ORDER.PRODUCT = PIXOUT.
  2. Upon receiving the API request, the system performs the following:
     1. Validates the debtor account and reserves funds.
     2. Creates the `PAYMENT.ORDER` in ‘AwaitingExtSubmit’ status.
     3. Returns a 200 success response with the Payment Order ID.

        If validation fails, no transaction is created and a 400 error is returned.
  3. After a positive acknowledgement from the external payment system, the transaction is booked in TPH (debit customer account and credit internal PIX clearing account) and the status is updated to ‘Complete’.
  4. If a negative acknowledgement is received, the reserved funds are reversed, the status remains ‘AwaitingExtSubmit’, and a 200 response with payment details is sent.
- Future-Dated PIX Outgoing (PIXFUTURE)

  The flow of this scenario is given below:

  1. Users can initiate future-dated PIX payments through API using PAYMENT.ORDER.PRODUCT = PIXFUTURE.
  2. The system performs the following:
     1. Validates the debtor account
     2. Creates the `PAYMENT.ORDER` in ‘AwaitingExtSubmit’ without reserving funds
     3. Returns a 200 success response.

        If the validation fails, no transaction is created and a 400 error is returned.
  3. Before execution, the user can amend or cancel the payments through API without reserving funds or generating TPH entries.
  4. On the execution date, the External payment system retrieves eligible payments and triggers execution using the Payment Order ID.
  5. Transact revalidates the debtor account, reserves funds, retains the status ‘AwaitingExtSubmit’, and switches the product from PIXFUTURE to PIXOUT (keeping the original reference).
  6. The system returns the 200 success response upon successful processing; else, a 400 error is returned if validation fails. Subsequent acknowledgement handling follows the same logic as same-day PIX Outgoing payments.

## PIX Incoming Payments

The following flow explains the processing for PIX Incoming payments in Transact:

1. The external payment system sends a valid API request to Transact, upon which a `PAYMENT.ORDER` transaction is initiated under the PIXIN product.
2. Transact determines the internal debit account based on configuration.
3. The debit account is derived from the Auto New Content setup in the `PAYMENT.ORDER` local version
4. The attached Validate routine overrides the default value if a PIXIN Suspense Account is configured in TPS.INTERNAL.CONFIGS.
5. Before posting, Transact performs validation on the identified internal debit account (status check) and on the beneficiary account (status and applicable restrictions). These validations ensure that both accounts are active and free from restrictions that would prevent transaction processing.
6. Upon successful validation, the `PAYMENT.ORDER` is created and submitted in ‘Complete’ status.
7. Accounting entries are generated by debiting the configured Internal PIX Suspense Account and crediting the beneficiary account. If any validation fails, the `PAYMENT.ORDER` is not created and no accounting entries are posted within Transact.

## PIX Intrabank Payments

The following flow explains the processing for PIX Intrabank payments in Transact:

1. To process a PIX intrabank transfer between two customer accounts within Transact, the External payment system sends a valid API request, and a `PAYMENT.ORDER` is initiated under the PIXINTRA product.
2. Since this is an intrabank transaction, the entire processing and settlement occur within Transact, without routing to the Central Bank’s Instant Payment System (SPI).
3. Before posting, Transact validates both the debit and credit accounts for status and applicable restrictions to ensure they are active and eligible for transaction processing. These validations are mandatory prior to creating the payment order.
4. Upon successful validation, the `PAYMENT.ORDER` is created and committed in ‘Complete’ status.
5. Accounting entries are generated by debiting the customer’s debit account and crediting the beneficiary’s credit account. If validation fails, the `PAYMENT.ORDER` is not created, and no accounting entries are generated within Transact.

## MED Return Request Payments

Payment Order processes same-day MED return requests through the Generic API (v7.4.0) using PAYMENT.ORDER.PRODUCT = MEDRETREQ. When the system receives the API request, it validates the debtor account, reserves funds, creates the PAYMENT.ORDER with the status ‘AwaitingExtSubmit’, and returns a 200 success response with the Payment Order ID. If validation fails, the system does not create a transaction and returns a 400 error.

Within 7 working days, the External Payment System provides the negative acknowledgement for the API request by sending the paymentOrderID in the request.

After the external payment system sends a positive acknowledgement, the system books the transaction in TPH (debit customer account and credit internal MED clearing account) and updates the status as ‘Complete’. If the external payment system sends a negative acknowledgement, the system reverses the reserved funds, retains the status as ‘AwaitingExtSubmit’, and returns a 200 response with the payment details.

## MED Return Payments

Payment Order processes MED return payments through the Generic local API (similar to v7.4.0) using PAYMENT.ORDER.PRODUCT = MEDRET. When the system receives the API request, it derives the Debit account internally from the local PAYMENT.ORDER version.  Optionally, the user can override the derived Debit account through the Validate API using the MEDRET Suspense Account configured in TPS.INTERNAL.CONFIGS. The system validates both Debit and Credit accounts (status and restrictions), does not reserve funds, creates the PAYMENT.ORDER directly with the status ‘Complete’, and returns a 200 success response with the Payment Order ID. If validation fails, the system does not create a transaction and returns a 400 error.

Since a MED return represents approved funds received from the external payment system, no acknowledgement cycle is required. After successful validation, the system immediately books the transaction in TPH (debit internal MED suspense/settlement account, credit customer account) and moves it to status 999 (Complete), and returns a 200 response with the payment details to the external payment system.

## PIX Saque Incoming Payments

The external payment system sends a valid API request to Transact, after which the system initiates a PAYMENT.ORDER transaction under the PIXSQIN product. Transact determines the internal debit account dynamically based on the configuration. The debit account is derived from the Auto New Content setup in the local PAYMENT.ORDER version and may be overridden through the attached Validate routine if a PIXSQIN Suspense Account is configured in TPS.INTERNAL.CONFIGS.

Before posting, Transact validates the identified internal debit account (status check) and the beneficiary customer account (status and applicable restrictions) to ensure both accounts are active and free from restrictions that would prevent transaction processing.

After successful validation, the system creates the PAYMENT.ORDER with the status ‘Complete’, generates a TPH transaction and moves it to 999, and posts accounting entries by debiting the configured Internal PIX Suspense Account and crediting the customer account. If validation fails, the system does not create a PAYMENT.ORDER and does not post accounting entries within Transact.
